# PennyWise Infrastructure (Terraform)

Provisions the API backend on **AWS ECS/Fargate** for `staging` and `prod`.
`dev` runs locally via `docker-compose` and has no cloud footprint.

## Layout

```
infra/
  shared/            # account-global, applied once (ECR repo)
  *.tf               # per-environment stack (VPC, ECS, ALB, DynamoDB, …)
  env/
    staging.tfvars
    prod.tfvars
```

The per-env stack uses **Terraform workspaces** (`staging`, `prod`) — one
codebase, separate state per environment. Table names are prefixed
`pennywise_<env>_`, mirroring `PENNYWISE_TABLE_PREFIX` in
`pennywise/api/db.py`. Terraform is the **source of truth for the DynamoDB
schema** — the `python -m pennywise.api.db --create` CLI is for local
dynamodb-local only.

## One-time bootstrap

Full walkthrough (fresh AWS account, IAM Identity Center setup, Google
Cloud Console): `docs/AWS_SETUP.md`. Summary:

1. **State backend** — `scripts/bootstrap-aws.sh` creates the S3 bucket
   referenced in `versions.tf` (`pennywise-tfstate`, versioned, encrypted,
   TLS-only). S3 native locking (`use_lockfile`) needs no DynamoDB lock table.

2. **Shared account-global resources** (ECR repo, GitHub OIDC role,
   CloudTrail):
   ```bash
   cd infra/shared
   terraform init && terraform apply
   ```
   Set the GitHub repo secret `AWS_DEPLOY_ROLE_ARN` to this apply's
   `deploy_role_arn` output, and create the `production` GitHub Environment
   with a required-reviewer rule (see `.github/workflows/`). No long-lived
   IAM user access keys are used anywhere — CI assumes the `pennywise-deploy`
   role via OIDC, scoped to exactly ECR push + ECS deploy + `iam:PassRole`
   on this project's own task/execution roles.

## Deploy an environment

```bash
cd infra
terraform init
terraform workspace select staging || terraform workspace new staging
terraform apply -var-file=env/staging.tfvars
```

Then populate secrets (created empty by Terraform — the service stays unready
until they exist):

```bash
aws secretsmanager put-secret-value --secret-id pennywise-staging-anthropic-api-key   --secret-string 'sk-ant-…'
aws secretsmanager put-secret-value --secret-id pennywise-staging-google-client-id     --secret-string '…'
aws secretsmanager put-secret-value --secret-id pennywise-staging-google-client-secret --secret-string '…'
# JWT_SECRET is generated by Terraform automatically.
```

Build + push the first image and point the service at it:

```bash
SHA=$(git rev-parse --short HEAD)
ECR=$(terraform output -raw ecr_repository_url)
docker build -t $ECR:$SHA .
aws ecr get-login-password | docker login --username AWS --password-stdin ${ECR%/*}
docker push $ECR:$SHA
terraform apply -var-file=env/staging.tfvars -var="image_tag=$SHA"
```

(After the first apply, CI handles image builds + rolling `ecs update-service`;
the service ignores `task_definition`/`desired_count` changes so Terraform and
CI don't fight.)

## Notes

- **Cost levers baked in:** single NAT gateway, free DynamoDB Gateway endpoint,
  Fargate Spot for staging, PAY_PER_REQUEST DynamoDB. See the root deployment
  plan for the full cost table.
- **`desired_count = 1`** until the SQS-backed job runner lands — in-memory
  background jobs don't survive more than one task.
- **HTTPS** requires `domain_name` + `route53_zone_id` in the tfvars; without
  them the ALB serves plain HTTP for bootstrap only.
